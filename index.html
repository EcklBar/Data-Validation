ğŸ—ï¸dbt Project Structure

+

You will develop a Data Mart in our dags-bi-eng repository. The project is organized into a three-layer architecture to ensure clarity and maintainability.

The Three Layers

1. Staging: The first layer of your model. Here you create an aggregated query based on the source tables.

2. Intermediate: The second layer. Here you add more complex manipulations and conditions to your query.

3. Mart: The final layer. This query will become a materialized table after you run the dbt run command.

Note: The Staging and Intermediate layers are not mandatory, but they are highly recommended for complex queries or models with multiple source tables.

Key Configuration Points

  • Materializations: How your model is built in the warehouse. The main types are:
    • Table: For dimension tables.
    • Incremental: For fact tables, loading data incrementally.
    • View: To create a view instead of a table.
  • Branching: Create a ticket under the EPIC BIP-600. Name your branch BIP-[issue_id]-your_branch_name.
  • YAML Generation: After creating or changing a model, generate its YAML file with:
    dbt run-operation generate_model_yaml --args '{"model_names": ["your_model_name"]}'
  • Testing: When testing with dbt run, specify a one-day range to reduce system load: dbt run -s your_model_name --vars '{"date_range": "1 day"}'
  • Naming Conventions:
    • fa_ for fact tables
    • di_ for dimension tables
    • ma_ for mart tables (joining facts and dims)
    • hi_ for historical tables

ğŸš€How Models are Triggered

+

Your dbt models are triggered by Apache Airflow. We use a tool called Cosmos to connect dbt and Airflow. Models run automatically once their source tables (DataX tables) are ready and release their triggers.

You can find available output triggers on this Kibana dashboard (last 14 days).

Note: Triggers from bq-outbrain-research.prod tables follow the pattern: table_name_is_ready_in_bq.

Adding a New Model to the Schedule

After developing your DM, you need to add an object to the dbt_models_data_prod template. For example:

{
    "table_name": "masr_source_referrer",
    "input_trigger": ["fast_section_traffic_d_is_ready_in_bq"],
    "input_trigger_type": BIEDbtInputTriggerType.HOURLY,
    "table_partition_type": BIEDbtPartitionType.DAILY,
    "table_type": BIEDbtTableType.MART,
}

Pull Request (PR) Workflow

  • Code Review: Have at least one team member review your SQL logic.
  • Open a PR: When opening a PR, include a screenshot confirming a successful run in the dev environment. If you created a fact or historical table, specify its retention period in the PR description.
  • Notify the Team: Share your PR in the #ask-bi-engineering channel for approval.

âœ…Summary - Your Workflow

+
  1. Develop your Data Mart on a local branch and test it. This stage creates your DM in the dev datasets.
  2. Create a new object in the Airflow DAGs template using the relevant source triggers.
  3. Open a Pull Request (PR) with all relevant files, a screenshot of your successful test run, and a short description.
  4. Send the PR in the #ask-bi-engineering Slack channel for code review and approval.
  5. After the PR is approved, merge it into the master branch.
Good luck, and don't hesitate to reach out to us!

âš–ï¸Data Mart or LookML?

+

The Main Principle

  • LookML: Better for small, lightweight tables or when queries are not heavy or frequent.
  • Data Mart: Better for large/heavy tables or when queries run frequently and need pre-aggregated data.

Breakdown by Case

Small table (<1TB) with partition expiration:
â†’ Use LookML. The data is automatically cleaned up.
Small table (<1TB) without partition expiration:
â†’ Use Mart. Ensures structure as the table may grow indefinitely.
Huge table (>5TB):
â†’ Always use a Mart. Direct queries are slow and costly.
Big table (1â€“5TB) with dashboards running multiple times a day:
â†’ Use a Mart. Avoids expensive, repeated queries on raw data.
Big table (1â€“5TB) with dashboards running once a day or less:
â†’ Use LookML. Low frequency doesn't justify the overhead of a Mart.
Note: These are general guidelines. A small table might still require a Mart if the logic is very complex or if queries consume too many resources.
